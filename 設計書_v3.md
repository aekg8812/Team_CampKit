# サボり癖クリア（v3）設計書

## 1. このバージョンの方針

v2「AI予言法廷」は1日限りのどよめき体験が中心でした。v3は**継続して使うほど記録が育つ**アプリへ進化させます。あわせて、ハッカソンで確実に動かすことを最優先に、外部サービス（Claude API・Firebase）への依存をスイッチで切り離せる構成にしました。

決定した方針は次の4点です。

第一に、データ保存は2方式を切り替えられます。既定はローカル保存（A案）で、Firebaseのキーを入れてスイッチをオンにしたときだけクラウド保存（C案）に変わります。普段の開発とデモはローカルで完結するため、キーが無くても全機能が動きます。

第二に、画像の証拠判定もスイッチ式です。既定はモック（常に受理）、発表のときだけClaude APIの画像判定を使います。

第三に、世界観（裁判所・予言・前科）は今回いったん外し、ニュートラルな表現（課題・成功・失敗・レベル）でシンプルに作りました。世界観の付与は時間が余ったときの後工程に回せます。

第四に、マイページのグラフは成功・失敗の割合を示す円グラフ1種に絞りました。テーマ別グラフは後回しです。

## 2. 用語の対応（v2 → v3）

世界観を外したことで、v2の用語を次のように置き換えています。判決は課題に、予言はカット、前科は失敗回数に、執行完了は成功に、尋問は質問に、裁判長のセリフはニュートラルなシステム表示になりました。

## 3. 画面フロー

初回利用時は、ログイン画面で新規登録し、ログイン後にサボり癖選択画面へ進みます。複数のサボり癖をチェックリストで選んで確定すると、マイページに着きます。

通常利用時は、ログイン後すぐマイページがホームになります。「今日の課題を始める」を押すと、選んだサボり癖の中からランダムで1カテゴリが選ばれ、そのカテゴリの質問10問に答えます。回答後はガチャ演出で課題が1つ決まり、課題画面に進みます。課題画面では中央に課題、その下にカウントダウンタイマー、さらに下に証拠提出（画像アップロードとコメント）と完了ボタンが並びます。完了するとその回が成功として記録され、結果画面を経てマイページに戻ります。タイマーが切れた場合は失敗として記録され、同じくマイページに戻ります。

## 4. サボり癖選択（最重要機能）

初回ログイン時、またはマイページからいつでも編集できます。選んだカテゴリだけが質問とガチャの対象になります。例えば運動不足を選んでいない人には運動の課題は出ません。「自分が向き合うと決めたこと」だけに集中させる設計です。

選べるカテゴリは、スマホ・SNS依存、部屋の汚さ、運動不足、先延ばし・積読、連絡の後回し、早起き、食生活の7種です。カテゴリの増減はデータファイル1つの編集で行えます。

## 5. レベル制とストリーク

カテゴリごとに難易度レベル（1〜3）と連続記録（ストリーク）を個別に持ちます。課題に成功するとそのカテゴリのレベルが1上がり、次回はより重い課題が出ます。失敗した場合はレベルを上げず据え置き、連続記録は0に戻ります。これにより「続けるほど手応えが増す」感覚と「途切れさせたくない」心理の両方が生まれます。

## 6. ログイン機能（最重要機能）

ユーザー名とパスワードで登録・ログインします。新規登録後はログイン画面に遷移し、あらためてログインします。パスワードを忘れた場合は、ユーザー名と新しいパスワード（確認用含む）で再設定できます。

ローカルモードでは認証情報もブラウザ内に保存します（パスワードは簡易ハッシュ。デモ用の簡易実装です）。Firebaseモードでは Firebase Authentication を使い、ユーザー名を内部的にメール形式へ変換して扱います。なお、ユーザー名運用のためメール送信を伴うパスワード再設定はFirebaseモードでは無効で、ローカルモードのみ対応します。

## 7. 証拠提出と画像判定（最重要機能）

課題画面で、写真を撮るか選んでアップロードできます。画像があればClaude APIに画像と課題内容を渡し、「常識的に見て妥当か」をゆるく判定させます。厳密な採点はせず、明らかに無関係・明らかにサボりでない限り合格とします。判定がグレーなときも合格扱いです。

写真が無い場合は、コメントを入力して自己申告で完了できます。判定に失敗したときも救済として受理します。これにより、証拠提出のハードルでデモが止まることを防ぎます。

繰り返しになりますが、この画像判定はスイッチが既定（オフ）のあいだはモックとして常に受理します。発表時にだけ本物のAPI判定を有効にします。

## 8. マイページ

総合成績として、成功・失敗の割合を円グラフで表示します。あわせてカテゴリ別の連続記録（現在・最高・レベル）と、直近のクリア履歴をリスト表示します。クラウド保存中はその旨も表示されます。

## 9. データ構造

ユーザーごとに、選択したサボり癖の配列、カテゴリ別のストリーク情報（現在・最高・レベル）、成功回数、失敗回数、履歴の配列を保持します。ローカルモードではブラウザのlocalStorageに、Firebaseモードでは Firestore の defendants コレクションに、同じ形で保存します。画面側は統一ストアだけを呼ぶため、どちらに保存されているかを意識しません。

## 10. 技術スタック

フロントは React（Vite）、スタイリングは Tailwind CSS、アニメーションは Framer Motion、効果音は Howler.js を使います。クラウド保存は Firebase（Authentication と Firestore）、画像判定は Claude API です。円グラフは外部ライブラリを使わずSVGで自作しています。デプロイは Vercel を想定しています。

## 11. ファイル構成

```
src/
├── App.jsx                  画面フロー全体の制御
├── main.jsx / index.css
├── data/
│   ├── habits.js            サボり癖カテゴリの定義
│   ├── questionsByHabit.js  カテゴリ別の質問（各10問）
│   └── tasksByHabit.js      カテゴリ別・レベル別の課題
├── store/
│   ├── index.js             統一ストア（A案/C案を振り分け）
│   ├── localStore.js        ローカル保存版
│   └── firebaseStore.js     Firebase版
├── lib/
│   ├── claude.js            画像判定（スイッチ式）
│   ├── firebase.js          Firebase初期化
│   └── sound.js             効果音
└── screens/
    ├── LoginScreen.jsx      ログイン・登録・再設定
    ├── HabitSelectScreen.jsx サボり癖選択
    ├── MyPageScreen.jsx     マイページ（円グラフ・記録）
    ├── QuestionScreen.jsx   質問10問
    ├── GachaScreen.jsx      課題決定ガチャ
    ├── TaskScreen.jsx       課題実行・証拠提出
    └── ResultScreen.jsx     成功/失敗
```

## 12. 起動とモード切り替え

普段の開発・デモでは、`.env.local.example` を `.env.local` にコピーするだけで動きます（中の値は空のままでよい）。`npm install` のあと `npm run dev` で起動します。

発表で本物の機能を見せたいときは、`.env.local` の `VITE_USE_API` や `VITE_USE_FIREBASE` を `true` にし、対応するキーを入れて、devサーバーを再起動します。
